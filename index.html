#Basic example

Passing the input element as a parameter to Tagify will transform it into a tags-component. Without any settings, the user will be allowed to create any tags they want, without a count limit.

The input has a pre-defined value which Tagify creates two from tags.

(Double-click on a tag to edit it)

HTML

<input name='basic' value='tag1, tag2'>

JAVASCRIPT

#text input

In this example:

  • Tagify instance with initial settings Object
  • Only allow tags from the whitelist to exist
  • Manually create initial whitelist from the original input's value
  • Simulate fetching whitelist (from a server) as the user is typing
  • "Remove all tags" by an extrenal button
  • Listen to various Tagify events
  • Un-listen to specific event (see onAddTag function)

Intro:

The input element is pre-ocupied with 4 tags. The last tag, "CSS", has the same value as the first tag, and will be removed, because the duplicates setting is set to true.

Note that whitelist & blacklist may also be set on to Tagify as data- attributes on the input tag itself as a list of tags seperated by the same delimeter defined in the Tagify's configuration. Default is comma (,)

As text is typed, the onInput Tagify event is fired, on each key stroke. Reseting the current whitelist and fecthing a new one (from mockAjax())

In real-life scenarios, a developer might wish to use different whitelists suggested to the user, according to the typed text, asusming the possibilities are too large to load at-once as the inital whitelist.

HTML

<input
  name='input'
  class='some_class_name'            // This class will be copied to the Tagify component
  placeholder='write some tags'      // This text will be shown after the last tag
  value='css, html, javascript, css' // initial values, as String (can also be an Array of Objects)
  data-blacklist='.NET,PHP'>         // list of invaild tags, which cannot be added

#textarea

In this example, the field is pre-ocupied with 3 tags, and last tag is not included in the whitelist, and will be removed because the enforceWhitelist setting flag is set to true

üö© This example is very interesting because it shows another layer of complexity - Films' names might include commas (,), and therefore it is very important to load initial values as Object Collection (Array of Objects) where each Object is proper JSON as shown in the example below, or else initial tags will not get parsed well if they include commas.

üö© Another important thing to keep in mind is that the default delimiters setting of Tagify is to split tags by commas, and when tags (might) contain commas inside them, this setting should be changed to null, so when a user is typing, for example, a movie name which contains commans, a tag will not be created as soon as comma was inputed.

HTML

<textarea name='tags2' placeholder='Movie names'>
    [{value:"The Good, the Bad and the Ugly"}, {"value":"The Matrix"}
</textarea>

#Mix text & tags

It is possible to configure Tagify to supoprt mixed content. A common example would be tagging people while writing a comment on a social website.

To allow mix content, simply pass the setting mode : 'mix'

In this example, there are a few Southpark character names and a few Simpsons ones. when the @ is typed, following 1 more character, suggestions dropdown will be shown for Southpark suggestion list and when starting with #, a Simpsons suggestion list

Note that tags can only be created if the value matches any of the whitelisted item's value.

When a textarea already has mixed-content and it wasn't pre-configured using a whitelist, the tags will ignored, so keep that in mind when populating a textarea with server data and expecting tags to "magically" appear. Whitelist must exist because not everything that starts with @ or # (or whatever was defined as pattern) is meant to be rendered as a tag.

Creating new tags:

To create new tags (ones not included in your whitelist collection), simply make sure to keep the enforceWhitelist setting as false, which is the default. After a new tags has been added, you should listen to the add and probably update your local state (Ex. server request).

HTML

<textarea name='mix'>
    [[{"id":101, "value":"cartman", "title":"Eric Cartman"}]] and [[kyle]] do not know [[homer simpson]] because he's a relic.
</textarea>

#Tags with properties

Some cases requires more control per-tag, for exmaple, sending a different value to the server than the textual value the user sees/entered. Another example, would be different colors for different tags or tags' groups. it's possible to add any number of properties per tag, the only constant is the value property which must be declared per-tag, and that will be the rendered text.

The propeties shown in the example below, declared in the allowedTags Array, will be transformed into HTML attributes for each tag element rendered from the allowed whitelist settings, so naturally some attributes are in the specs and the made-up ones should technically be prefixed with data-.

With this level of control a lot can be done via css.
For the example below, the first two tags have a readonly property and that will enable the CSS to target that and hide the × button and disable the hover state.

HTML

<input name='tags3-1' placeholder="Try to add tags from the list">

#Readonly mode

If the original input field has a readonly attribute, then, via CSS, there will be no way of manually adding tags because the inline contenteditable element will be hidden.

HTML

<input name='tags4' readonly value='tag1, tag 2, another tag'>
